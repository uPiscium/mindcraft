# Component 1.2: Runtime Task Pool Integration - 実フロー接続

## 目的

`MindcraftRuntime` に載ったタスクプールを、実サーバ上で動くエージェントの実行フローへ接続する．
この段階では手動APIではなく、実際の起動・実行・停止のライフサイクルに合わせて自動連携する．

## 接続ステップ

### 1. タスク供給元の確定

- 起動時にどこからタスクを登録するかを決める
- 少なくとも次のいずれかを受けられるようにする
  - CLI 引数
  - profile 設定
  - task file
- 登録時に `depends_on` と `priority` を保持する

### 2. エージェント起動時のタスク初期化

- `start_agent_process()` / `create_agent_process()` の流れで、対象エージェントに紐づくタスクを runtime に登録する
- エージェントごとのタスク範囲を明示する
- 登録済みタスクの重複投入を防ぐ

### 3. 実行ループへの acquire 接続

- 行動開始前に `acquire_task(requester_id)` を呼ぶ
- 取得成功時のみ、そのタスクを実行コンテキストへ渡す
- 取得失敗時は、待機・再試行・終了のいずれかに分岐する

### 4. 実行完了時の yield 接続

- タスク実行完了後に `yield_task(requester_id, task_id, reason)` を呼ぶ
- 正常終了時と失敗時で `reason` を分ける
- 例外・中断時も必ず返却経路を通す

### 5. 停止・切断時の解放

- エージェント停止時にロック中タスクを解放する
- プロセス終了時に `LOCKED` のタスクが残らないようにする
- 異常終了時も回収処理を行う

### 6. 状態可視化

- `list_tasks()` の結果を確認できるようにする
- エージェントの状態表示にタスク ID と状態を出せるようにする
- 依存待ち・実行中・返却済みを区別できるようにする

## 仕様テスト

### `tests/test_runtime_task_pool_integration.py`

- エージェント起動時にタスクが登録される
- 実行開始前に `acquire_task()` が呼ばれる
- 実行完了後に `yield_task()` が呼ばれる
- 停止時にロック中タスクが解放される
- 依存未完了タスクは取得されない

### `tests/test_agent_process.py`

- 起動・停止のライフサイクルでタスク解放が壊れない
- 例外時でも返却経路が通る

## 非対象

- タスク定義フォーマットの再設計
- タスクの永続化
- 分散キュー化
- 優先度アルゴリズムの変更
