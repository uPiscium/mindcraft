# Component 1.1: Runtime Task Pool - 最小接続仕様

## 目的

既存の `CentralTaskCoordinator` を、`mindcraft_py.runtime.MindcraftRuntime` から手動で扱えるようにする．
この段階では自動実行ループには接続せず，公開APIとしての整合性だけを固める．

## 公開インターフェース

### `register_task(task)`

- 役割: タスクを runtime のタスクプールへ手動登録する
- 入力: `task` はタスクエンティティまたは同等の辞書表現
- 出力: 登録済みタスクの表現
- 振る舞い:
  - 同一 `id` が既に存在する場合は上書きする
  - `AVAILABLE` / `LOCKED` などの状態は保持する

### `list_tasks()`

- 役割: 現在のタスク一覧を返す
- 出力: タスクの配列
- 振る舞い:
  - 読み取り専用
  - 順序は登録順を維持する

### `acquire_task(requester_id)`

- 役割: 条件に合う `AVAILABLE` タスクをアトミックに取得する
- 条件:
  - `state == AVAILABLE`
- 振る舞い:
  - 優先度順で 1 件だけ返す
  - 取得時に `LOCKED` へ遷移する
  - `lock_metadata` に取得者と時刻を記録する
- エラー:
  - 条件に合うタスクがない場合は Conflict 系エラー

### `yield_task(requester_id, task_id, reason)`

- 役割: ロック中タスクを返却する
- 条件:
  - `task_id` が `requester_id` によりロックされていること
- 振る舞い:
  - `history` に `reason` を追記する
  - `state` を `AVAILABLE` に戻す
  - `lock_metadata` を解除する
- エラー:
  - 所有者不一致は拒否する
  - 存在しない `task_id` は拒否する

## タスク入力元

- 手動登録のみ
- 起動時の自動投入は行わない
- 設定ファイルや profile からの初期ロードはこの段階では対象外

## 依存関係

- `Task Entity` は `depends_on: [task_id, ...]` を持てる
- 依存関係はタスク ID の配列として表現する
- 依存タスクが `COMPLETED` のときだけ取得対象になる

## 仕様テスト

### `tests/test_runtime_task_pool.py`

- `register_task` が登録済みタスクを保持する
- `list_tasks` が登録済みタスクを返す
- `acquire_task` が条件一致タスクを `LOCKED` にする
- `acquire_task` が優先度の高いタスクを選ぶ
- `acquire_task` が該当なしで Conflict 系エラーになる
- `yield_task` が所有者以外を拒否する
- `yield_task` が履歴を追記して `AVAILABLE` に戻す
- 同一タスクの二重取得が起きない

## 非対象

- エージェント実行ループへの自動接続
- タスクの永続化
- 設定ファイルからの自動ロード
- UI やコマンド経由の公開

## 【実装メモ】

- `mindcraft_py.runtime.MindcraftRuntime` に `register_task` / `list_tasks` / `acquire_task` / `yield_task` を追加済み．
- 内部では `mindcraft_py.task_coordinator.CentralTaskCoordinator` を共有し，`shutdown()` でクリアする．
- 回帰テストは `tests/test_runtime_task_pool.py` にある．
